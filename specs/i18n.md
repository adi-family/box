# `@box/i18n` Plugin Spec

Status: **Draft v0.4** — working document.

This spec documents `@box/i18n` as **one implementation** of Box's
i18n service contract. Other plugins may provide the same service with
different body grammars (Fluent-backed, gettext-backed, Crowdin-backed,
embedded-table, etc.). The cli plugin and any other consumer asks the
manager for "whoever provides i18n" — not for this package specifically.

Layered on top of the [core Box language](../SPEC.md), which uses
Compose-style block syntax: `kind("name") { ... }`.

---

## 1. What Box core defines (not this plugin)

Two things — and only two — about i18n live in core:

1. **The `t(...)` value form.** A plugin-whitelisted named function call
   accepted in fields whose schema declares them translatable. See core
   §7.
2. **The i18n service contract.** Any plugin that claims to provide
   i18n must implement `resolve(key, locale, params) -> string`.

Everything else — locale config, file format, load strategy, fallback,
hot reload — is the plugin's call.

---

## 2. Plugin

- Package: `@box/i18n`
- Kinds claimed: `i18n`
- Service provided: `i18n` (so consumers reach this plugin via
  `plugins.get_provider("i18n")`)
- Whitelisted functions contributed: `t(key, ...named-params)` in fields
  schema-marked as accepting translatable values. The schema marks `t`
  as requiring the **default instance** of the `i18n` kind (core §2) —
  the parser refuses to compile a file using `t(...)` unless that file
  directly imports the file containing the `i18n { }` block.
- Loaded via `use "@box/i18n"`.

---

## 3. The `i18n` kind (this plugin's body grammar)

The body grammar below is **this plugin's** choice. Another i18n
provider could expose a different body shape.

### Shape

```box
use "@box/i18n"

i18n {
  locales = ["<locale-id>", ...]   // declared supported locales
  default = "<locale-id>"          // optional, must be in `locales`
}
```

### Body schema

| Field | Type | Required | Notes |
|---|---|---|---|
| `locales` | array of strings | yes | List of supported locale identifiers (BCP-47 style — `en`, `ru`, `de-CH`, etc.) |
| `default` | string | no | Fallback locale; must be present in `locales`. If omitted, the runtime / build target picks. |

No fields declaring file paths, frameworks, file formats, or eager-load
config. Everything else is configured outside Box — in this plugin's
own config file, environment, or CLI flags — at the plugin's discretion.

### Lazy by design

The plugin must load translation data **per-locale, on demand**. A
project declaring 200 locales pays for 1 at runtime (the requested
locale, plus the fallback chain if any).

### Usage rules

- **Nameless form is the project default.** Write `i18n { ... }`.
  At-most-one-default-per-kind is a core parser rule (§2).
- **Named instances are rejected** by this plugin's `validate()` —
  `i18n("emails") { ... }` parses, but the plugin returns an error
  because multiple i18n configurations aren't meaningful here.
- **Files using `t(...)` must directly import** the file containing the
  `i18n { ... }` block. Parser-enforced via the function's schema
  declaration that it depends on the `i18n` default instance.

## 4. The `t(...)` function call

This plugin gives meaning to the `t(...)` value form (declared in core
§7 as a plugin-whitelisted named function call):

```
t(<key>, <name> = <value>, ...)
```

- **First positional arg**: a string literal — the translation key path.
- **Named args (zero or more)**: parameter map passed to the resolver.

### Examples

```box
description = t("cli.init.description")

description = t(
  "cli.main.description",
  version = package.box.version,
  edition = "2026",
)

label = t("forms.email.label")
```

### Where it's accepted

Only in fields whose plugin kind schema marks them as accepting a
translatable value. Field schema kind: `translatable_string` (accepts
either a plain string or a `t(...)` call). Plugins authoring CLI / UI /
docs surfaces mark their human-facing fields this way.

### Parameter values

Parameters can be any value form Box supports: string, number, bool,
identifier, cross-block field ref (`package.box.version`), nested
function call (a `t(...)` whose result is interpolated), array.

## 5. RPC service contract

These methods are the **service contract** that any i18n provider must
implement (not just `@box/i18n`). The cli plugin (and any consumer)
calls them via the cross-plugin manager.

### `resolve`

```
resolve(key: string, locale: string, params: { [name]: string }?)
  → ResolvedString
```

Look up `key` for `locale`, interpolate `params`, return a localized
string.

```
ResolvedString {
  ok:       boolean
  value:    string?       // present when ok = true
  fallback: boolean       // true if resolved via fallback chain
  error:    string?       // missing key, malformed selector, etc.
}
```

### `locales`

```
locales() → string[]
```

Returns the locale identifiers this provider knows about.

### `keys` (optional)

```
keys(locale: string?) → string[]
```

All known keys (across all locales if `locale` is omitted, else for the
given locale). Used by tooling for completion and "missing translation"
linting. Implementations may decline (`null` response).

## 6. Open questions

These are **plugin design** questions, not language questions:

1. **Resolution timing.** Codegen-time (per-locale baked output) vs.
   runtime (single output reads translation source at run). User
   direction: leans dynamic / runtime.
2. **Translation source.** Files (Fluent, gettext, JSON), remote
   service (Crowdin, Locize), embedded blobs, hybrid. Plugin's
   config — possibly per-locale.
3. **Fallback chain.** Linear (`default` only) vs. transitive
   (`de-CH` → `de` → `default`). v0.2 picks single fallback to `default`.
4. **Missing-key behavior.** Return the key literally? Return an error
   per the `ResolvedString.error` field? Configurable per-target?
5. **Key extraction.** A `box i18n extract` CLI subcommand that scans
   all `t(...)` calls and emits an empty translation skeleton.
6. **Pseudo-locale generation.** Auto-generate `xx-pseudo` from
   `default` for translation-coverage testing.
7. **Selectives.** ICU MessageFormat is what most frameworks use for
   plurals/gender/choice. Confirm we standardize on it as the param
   semantics across implementations.

## 7. Example

```box
// box/box/i18n.box

use "@box/i18n"

i18n {
  locales = ["en", "ru", "de"]
  default = "en"
}
```

```box
// box/box/cli.box

use "@box/cli"
use "@box/i18n"
import "./i18n.box"   // required: wires up the i18n singleton

cli("main") {
  command("init") {
    description = t("cli.init.description")
    arg("name") {
      description = t("cli.init.arg.name")
      required    = true
    }
  }
}
```

At codegen time (or runtime, depending on the chosen strategy), the cli
plugin resolves `t(...)` values by calling the i18n provider:

```
i18n.resolve("cli.init.description", "en", null)
  → { ok: true, value: "Create a new Box project", fallback: false }
```

Where the translation strings actually live, and in what format, is
**this plugin's call** — outside Box's spec.
